Структуры позволяют нам хранить комплексные данные и представляют способ организовать данные под определённый класс операций.

Напомню, что примитивами мы считаем простые типы: числа, строки и специальные логические значения или отсутствия значения. Например в Python:

age = 30            # int  — целое число

От способа организации данных зависит, какие операции будут быстрыми, какие — медленными, а какие — вообще неудобными. Посмотрим на два примера, где мы решаем одну и ту же задачу, используя только примитивы и массив.

```
anna = "Alice"

print("Names:", anna, bob, carol)
```

 С массивом мы можем легко добавлять новые имена:
 ```
 names = ["Alice", "Bob", "Carol"]
 
 print("Names: ", names)
 ```

В этом видео мы разберём три базовые структуры данных, каждая из которых имеет уникальные возможности:
- Массив - предоставляет доступ по индексу
- Хэш-таблица - быстрый поиск по ключу
- Множество является частью хэш-таблицы и определяет уникальность входящих элементов

Для каждой структуры обсудим, как они устроены на уровне абстракции, какие операции поддерживают лучше всего и как они представлены в популярных языках программирования: Python, Go, C++ и Rust.

## Массивы: линейная организация данных (5 мин)

Начнём с самой базовой и, пожалуй, самой интуитивной структуры данных — массива.

>Массив — это структура, которая хранит элементы последовательно и позволяет обращаться к ним по индексу. То есть у каждого элемента есть фиксированная позиция: нулевой элемент, первый, второй и так далее.

Если говорить абстрактно, массив нужен тогда, когда мы хотим хранить набор однотипных или логически связанных значений в определённом порядке. Например, список оценок, координаты точек, последовательность команд, массив чисел, список пользователей и т. д. Для всех этих случаев нам может быть важен порядок их обработки. В случае с координатами мы можем восстановить маршрут, а пользователи могут храниться в алфавитном порядке.

Главная идея массива в том, что элементы расположены последовательно, другими словами они хранятся подряд. Именно поэтому операция доступа по индексу обычно очень быстрая. Если нам нужен элемент с индексом 5, мы не перебираем все предыдущие элементы, а сразу обращаемся к нужной позиции. Это гарантируется последовательным хранением элементов в памяти и адресной арифметикой.

Когда задача требует много раз читать элементы по конкретному индексу, вы скорее всего будете использовать массив. Например, если мы хотим быстро взять середину последовательности, поменять местами два элемента или пройтись по данным слева направо.

Как и у любого другого решения, у массива есть и ограничения: вставка или удаление в середине массива будет требовать накладных расходов. Массив хранит элементы последовательно, и если мы вставляем новое значение в середину, все элементы справа нужно сдвинуть, чтобы освободить место. А при удалении, наоборот, элементы нужно подтянуть влево, чтобы не осталось “дырки”. Особенности хранения массива в памяти приводят к тому, что мы должны перезаписать весь массив. Однако вставка в конец массива уже не такая затратная. Разберём эти особенности на примерах:

Вернёмся к примеру с именами:
`names = ["Alice", "Bob", "Carol"]`
Допустим, мы хотим вставить `"Dave"` между `"Alice"` и `"Bob"`. Нужно сдвинуть всё, что стоит правее. Поэтому массив хорош не для любых операций, а для вполне конкретного класса задач. В памяти компьютера массив занимает **непрерывный** участок памяти: элементы лежат подряд, поэтому по индексу мы можем вычислить адрес элемента простой формулой “базовый адрес + индекс × размер элемента”. Но из-за этой же непрерывности нельзя “вставить” новый элемент в середину, просто вклинив его между уже существующими значениями. Нужно либо сдвинуть часть элементов внутри уже выделенного блока, либо, если места не хватает, создать новый массив большего размера, скопировать туда все элементы и начать работать уже с ним. Именно поэтому вставка и удаление в середине массива стоят дороже, чем чтение по индексу или добавление в конец. 

Чтобы не перевыделять память при каждом добавлении в конец, реализация обычно имеет запас памяти. Когда массив расширяется, создаётся новый, более крупный массив и все элементы копируются — это дорогая операция, поэтому выделенное место создаётся с запасом, большим, чем текущее количество элементов. Поэтому повторное добавление не потребует переписывания массива на новое место. Таким образом **амортизированная** стоимость добавления в конец остаётся близкой к `O(1)`.

В итоге массив эффективен, когда основная работа — это чтение, последовательный обход, доступ по индексу и добавление в конец. Мы посмотрели на абстракцию массива со стороны computer science, а вот реализация в языках программирования всегда имеет свои особенности.

В Python под массивом мы понимаем список - `list`.  
`list` ведёт себя как динамический массив, то есть нам не нужно думать о памяти, изменение размера происходит "под капотом". К элементам списка можно обращаться по индексу, можно добавлять и удалять элементы и итерироваться по массиву. Итерация означает последовательный доступ ко всем элементам. Важной особенностью в Python так же является возможность хранения в одном массиве разных типов данных, обычно массивы позволяют хранить только объекты одного типа.

Простой пример на Python:
```python
# Python — массив (list)

numbers = [10, 20, 30, 40]

print(numbers)       # [10, 20, 30, 40]
print(numbers[1])    # 20

numbers.append(50)   # добавление в конец
numbers.insert(1, 15)  # вставка по индексу

print(numbers)       # [10, 15, 20, 30, 40, 50]
```

В Go есть массивы фиксированной длины, но в реальной разработке гораздо чаще используют `slice`. Слайс — это представление над массивом, то есть не просто набор элементов сам по себе, а структура, которая работает "поверх" существующего массива. Слайс скрывает реализацию расширения массива, значительно упрощая жизнь разработчика. Слайсы имеют две величины: `len`(от length - длина) и `cap`(от capacity - вместимость). `len` показывает, сколько элементов сейчас находится в слайсе, а `cap` — сколько элементов можно вместить в текущую выделенную область до перераспределения памяти.

Пример на Go:
```go
nums := []int{10, 20, 30, 40}

fmt.Println(nums)   // [10, 20, 30, 40]
fmt.Println(len(nums)) // 4
fmt.Println(cap(nums)) // 4

nums = append(nums, 50)
fmt.Println(nums)   // [10, 20, 30, 40, 50]
fmt.Println(len(nums)) // 5
fmt.Println(cap(nums)) // 8
```

В C++ массив представлен как `std::vector`.  
Согласно документации, `std::vector` — это *sequence container*, который инкапсулирует *dynamic size array*. То есть это динамический массив, который хранит элементы последовательно, даёт доступ по индексу и автоматически управляет расширением памяти.

Пример на C++:
```cpp
#include <vector>
#include <iostream>

int main() {
    std::vector<int> nums = {10, 20, 30, 40};

    std::cout << nums << std::endl; // [10, 20, 30, 40]

    nums.push_back(50);
    nums.insert(nums.begin() + 1, 15);
}
```

В Rust роль динамического массива выполняет `Vec<T>`.  
Документация описывает `Vec<T>` как contiguous *growable array type*, то есть непрерывный, растущий массив. Rust даёт безопасную модель работы с памятью, поэтому многие операции сопровождаются более строгими правилами доступа к данным.

Пример на Rust:
```rust
fn main() {
    let mut nums = vec!;[6][7][8][5]

    println!("{}", nums); // [6, 7, 8, 5]

    nums.push(50);
    nums.insert(1, 15);
}
```

Для каких задач массив подходит лучше всего:
Во-первых, когда нужен быстрый доступ по индексу
Если задача регулярно обращается к элементам по позиции, массив — естественный выбор.

Во-вторых, когда важен порядок элементов
Массив хорошо представляет упорядоченные последовательности: список значений, временной ряд, входные данные задачи, маршрут, набор команд.

В-третьих, когда нужен последовательный обход значений
Это может быть линейный или бинарный поиск по отсортированным данным, сортировки, работа с указателями, префиксные суммы и многие другие алгоритмы.

Итак, массив — это структура для последовательного хранения элементов с быстрым доступом по индексу.  Она отлично подходит для упорядоченных данных и последовательной обработки, но не для частых вставок и удалений в середине.

## Хэш-таблицы: доступ по ключу (5 мин)

Перейдём ко второй структуре данных — хэш-таблице. Если в массиве у нас есть доступ к элементам по индексу, то в хэш-таблице быстрый доступ к данным осуществляется по **ключу**. Хэш-таблица хранит пары «ключ-значение» и позволяет быстро находить связанное с ключом значение.

Хэш-таблицу часто называют словарём, это отлично описывает пример использования.  У нас есть слово — это ключ, и есть его определение — это значение. Когда мы ищем определение, мы не хотим просматривать весь список подряд, как в массиве, а хотим сразу попасть к нужному значению. Хэш-таблица как раз и решает эту задачу: вместо индекса используется ключ, а сама структура берёт на себя работу по вычислению того, где именно хранить значение.

Как это происходит внутри на верхнем уровне?  
Когда мы кладём пару «ключ-значение» в хэш-таблицу, ключ сначала проходит через специальную хэш-функцию. Эта функция берёт ключ и преобразует его в число — хэш-код. Этот хэш-код используется, чтобы определить позицию во внутреннем хранилище, обычно это какой-то массив или набор так называемых бакетов.

То есть в отличие от массива, где индекс задаём мы сами и он равен порядковому номеру элемента в списке, в хэш-таблице индекс получается как результат работы хэш-функции над ключом. Можно себе представить, что хэш-таблица - это массив, где вместо индексов используются наши ключи, а в ячейках лежат значения. После того, как мы добавили ключу его значение, мы можем получить это значение используя тот же ключ: хэш-функция вычисляет хэш от ключа, определяет позицию и смотрит, какое значение лежит в соответствующем месте.

Основное достоинство хэш-таблицы в быстрых операциях с содержимым. Асимптотически **средняя** сложность основных операций (вставка, поиск, удаление) близка к `O(1)`. *Определение асимптотики О большой вы можете посмотреть в отдельном уроке по ссылке.* На практике любая реализация обеспечивает такую асимптоматику в **среднем**: скорость зависит от качества хеширования и того, как таблица справляется с коллизиями.

> Коллизия — это ситуация, когда два разных ключа дают одинаковый хэш-код и попадают в одну и ту же позицию внутреннего массива или бакета. Коллизии неизбежны, поскольку размер хранилища для ключей ограничен. Обычно хэш-таблица  обрабатывает коллизии одним из двух вариантов:
> 1. либо хранит несколько элементов в одном бакете (например, списком или другим способом)
> 2. либо использует более сложные схемы размещения, чтобы разнести элементы по разным позициям.

Мы не будем вдаваться в детали конкретных стратегий, так как это тема для отдельного урока, зафиксируем сам факт:  
"Так как разные ключи могут попасть в одну и ту же позицию реализация хэш-таблицы должна иметь механизм разрешения коллизий. От выбранной реализации зависят производительность и потребление памяти."

Теперь посмотрим, как эта абстракция реализована в популярных языках.

В Python это структура `dict`. Она хранит пары «ключ-значение», позволяет брать значение по ключу, добавлять новые пары, обновлять существующие и проверять наличие ключа. Структура использует хэширование и внутренний массив для хранения элементов, но все детали скрыты за удобным интерфейсом.

Пример на Python:
```python
ages = {
    "Анна": 28,
    "Борис": 32,
    "Вера": 25,
}

print(ages["Борис"])       # 32

ages["Глеб"] = 30          # добавление новой пары
print("Анна" in ages)      # проверка наличия ключа
```
Особенность реализации в Python в том, что порядок элементов в dict сохранятся, это редкая особенность, ведь основная цель — скорость доступа по ключу, а не сохранение порядка, как в массиве. Реализации на других языках не гарантируют такое поведение.

В Go в качестве реализации хэш-таблиц есть тип `map`. В официальной документации по языку написано, что `map` — это hash table, и операции поиска и вставки рассчитаны на постоянное среднее время.

Пример на Go:
```go
ages := map[string]int{
    "Анна": 28,
    "Борис": 32,
    "Вера": 25,
}

fmt.Println(ages["Борис"])     // 32

ages["Глеб"] = 30             // добавление новой пары

value, ok := ages["Анна"]     // проверка наличия
if ok {
    fmt.Println(value)
}
```

В C++ для хэш-таблиц используется `std::unordered_map`.  
Это контейнер, который хранит пары «ключ-значение» и использует хэширование для быстрого доступа. Документация так же указывает, что поиск, вставка и удаление элементов имеют в среднем постоянную сложность \(O(1)\).

Пример на C++:

```cpp
#include <unordered_map>
#include <iostream>
#include <string>

int main() {
    std::unordered_map<std::string, int> ages = {
        {"Анна", 28},
        {"Борис", 32},
        {"Вера", 25}
    };

    std::cout << ages["Борис"] << std::endl; // 32

    ages["Глеб"] = 30; // добавление новой пары

    if (ages.contains("Анна")) {
        std::cout << ages["Анна"] << std::endl;
    }
}
```

В Rust аналогичная структура — это `HashMap` из модуля `std::collections`.  
`HashMap` хранит пары `K, V`, использует хэш-функцию, операции доступа тоже рассчитаны на среднюю постоянную сложность.  

Пример на Rust:

```rust
use std::collections::HashMap;

fn main() {
    let mut ages = HashMap::new();

    ages.insert(String::from("Анна"), 28);
    ages.insert(String::from("Борис"), 32);
    ages.insert(String::from("Вера"), 25);

    if let Some(age) = ages.get("Борис") {
        println!("{}", age); // 32
    }

    ages.insert(String::from("Глеб"), 30);
}
```

В каких сценариях хэш-таблицы являются лучшим выбором?

Во-первых, при необходимости иметь быстрый поиск по идентификатору.
Если у нас есть сущности с уникальными ключами — пользователи по логину, товары по коду, записи по ID, — хэш-таблица позволяет получить доступ к данным по ключу за среднее постоянное время, без полного обхода коллекции. Однако, если ключом является постоянно-растущее положительное число вы можете использовать массив, он будет работать быстрее и занимать меньше памяти.

Хэш-таблицы часто используются для хранения конфигураций, таблиц маршрутизации, кэшей результатов вычислений, индексов по ключам в приложениях и других структур, где важен быстрый доступ к данным по ключу.

Во-вторых, подсчёт частот.
Хэш-таблица отлично подходит для решения популярной задачи, когда нужно посчитать сколько раз каждое слово встречается в тексте. Другие примеры: сколько раз встречается каждая цифра, символ, событие и т.д.
Хэш-таблица за счёт быстрого доступа позволяет эффективно увеличивать счётчик для ключа, который соответствует конкретному значению.

Во многих алгоритмических задачах нам нужно либо быстро проверять, есть ли в наборе элемент с заданным значением, либо хранить вспомогательные данные по ключу. Хэш-таблица позволяет обеспечить сложность `O(n)` или сделать решение проще в реализации.

Итак, хэш-таблица — это структура для организации данных по ключу с быстрым средним доступом. Используйте, когда нужно связывать ключи и значения, быстро проверять наличие элементов и считать частоты.
## Множества: уникальность и принадлежность (5 мин)

Следующая структура — множество. Если массив отвечает на вопрос “какой элемент лежит на позиции i”, а хэш-таблица на вопрос “какое значение связано с этим ключом”, то множество отвечает на вопрос: “есть ли этот элемент в наборе?”.

С точки зрения абстракции множество — это структура, которая хранит **только уникальные элементы**. Если вы добавите один и тот же элемент несколько раз, он всё равно будет присутствовать в единственном экземпляре. Само множество можно рассматривать со стороны математики и со стороны Computer Science.

Со стороны математики множество — это понятие представляющее собой набор, каких-либо объектов — **элементов** этого множества. Про множество можно сказать, что два множества равны тогда и только тогда, когда содержат в точности одинаковые элементы. 

В computer science множество - это структура данных, которая используется для эффективной проверки находится ли элемент среди уже добавленных. Зная определение хэш-таблицы и массива мы можем утверждать, что хэш-таблица состоит из двух частей:
- множества ключей;
- и ассоциативного массива значений, привязанных к этим ключам.

Если взять хэш-таблицу и отбросить из неё значения, оставив только ключи, получится структура, которая хранит набор уникальных ключей и позволяет эффективно определять, есть ли тот или иной ключ в этой структуре. Это и есть типичная реализация множества в современных языках: **hash set** — множество на основе хэш-таблицы.

Как и в хэш-таблице, в множестве, операции добавления элемента, проверки принадлежности и удаления в среднем гарантируют асимптотическую сложность `O(1)`. Таким образом, множество — отличный инструмент для задач, где нужно быстро проверять, встречался ли элемент ранее, и гарантировать уникальность.

Посмотрим на несколько сценариев использования.

Во-первых, **удаление дубликатов**.  
Если у нас есть список значений, и нужно оставить только уникальные, самый простой способ — пройтись по этим значениям и добавлять их в множество. Во множестве один и тот же элемент представлен только один раз, повторное добавление будет проигнорировано, поэтому на выходе мы получим набор уникальных значений.

Во-вторых, **проверка, встречался ли элемент раньше**.  
Популярное решение в алгоритмах: мы итерируемся по данным и хотим быстро понять, видели ли уже это значение. Например, ищем первый повторяющийся элемент или проверяем, содержит ли массив дубликаты. Здесь множество служит коллекцией уже просмотренных элементов — операции `add` и `contains` работают быстро, и не нужно каждый раз перебирать весь массив.

К этому же типу задач можно отнести алгоритмы, где мы отмечаем объекты как `visited`. Например, в графовых алгоритмах и поиске по состояниям объектами могут быть вершины графа, состояния в поиске по пространству, комбинации параметров. Множество позволяет эффективно отвечать на вопрос был ли обработан этот объект ранее и пропускать его.

В-третьих, **операции над множествами**: пересечение, объединение, разность.
Математические операции над множествами позволяют нам решать множество задач:
- пересечение — элементы, которые есть и в первом, и во втором множестве;
- объединение — все элементы из обоих множеств;
- разность — элементы, которые есть в первом множестве, но нет во втором.  
Эти операции встречаются в прикладных задачах: фильтрация пользователей по нескольким признакам, поиск общих элементов в наборах, вычисление симметрической разности и так далее. Реализация множества предлагает удобные интерфейсы для таких операций.

Посмотрим на реализации в конкретных языках.

В Python есть встроенный тип `set`, который реализован на основе хэш-таблицы. Он хранит уникальные элементы и обеспечивает эффективное выполнение операций добавления, удаления и проверки принадлежности.

Пример на Python:
```python
visited = set()

visited.add("Анна")
visited.add("Борис")
visited.add("Анна")  # повторное добавление, множество не меняется

print("Анна" in visited)   # True
print("Вера" in visited)   # False
```

В Python `set` хранит только ключи, а не пары “ключ-значение”, как `dict`. Реализация как у хэш-таблицы: элементы распределяются по бакетам с помощью хэш-функции.

В Go отдельного встроенного типа `set` нет.  
Компилятор Go позволяет использовать переменные без значений (они будут равны нулевым значениям для каждого типа) благодаря этому идиоматичный способ реализовать множество — использовать `map[T]bool` или `map[T]struct{}`. Ключи хэш-таблицы будут элементами множества, а значения заполнены значениями по умолчанию, это не снижает эффективность и соответствует правилу zero cost abstractions языка.

Пример на Go с `map[string]struct{}`:

```go
visited := make(map[string]struct{})

visited["Анна"] = struct{}{}
visited["Борис"] = struct{}{}
visited["Анна"] = struct{}{} // повторное "добавление"

_, ok := visited["Анна"]      // проверка принадлежности
fmt.Println(ok)               // true

_, ok = visited["Вера"]
fmt.Println(ok)               // false
```

> Тип `struct{}` часто выбирают для экономии памяти, потому что он не занимает места.

В C++ для множеств на основе хэш-таблицы есть контейнер `std::unordered_set`.  
Документация описывает его как контейнер, который хранит уникальные элементы и использует хэширование для быстрого доступа, обеспечивая среднюю сложность `O(1)` для операций вставки, поиска и удаления.

Пример на C++:
```cpp
#include <unordered_set>
#include <iostream>
#include <string>

int main() {
    std::unordered_set<std::string> visited;

    visited.insert("Анна");
    visited.insert("Борис");
    visited.insert("Анна"); // второй раз не добавится

    std::cout << visited.contains("Анна") << std::endl; // 1 (true)
    std::cout << visited.contains("Вера") << std::endl; // 0 (false)
}
```

Отдельно упомянем `std::set`, который обычно реализован на дереве поиска (например, красно-чёрном дереве). Он хранит уникальные элементы и обеспечивает упорядоченность при логарифмической сложности операций `O(log n)` что хуже чем средняя `O(1)`, как `unordered_set`. Это хороший пример того, что одна и та же абстракция, в нашем случае “множество” может быть реализована по-разному — через хэш-таблицу или через дерево.

В Rust для множеств есть `HashSet` в `std::collections`.  
Это коллекция уникальных элементов, реализованная на основе хэш-таблицы. Как и в случае `HashMap`, операции вставки и проверки принадлежности обеспечивают среднюю постоянную сложность.

Пример на Rust:
```rust
use std::collections::HashSet;

fn main() {
    let mut visited: HashSet<String> = HashSet::new();

    visited.insert("Анна".to_string());
    visited.insert("Борис".to_string());
    visited.insert("Анна".to_string()); // дубликат игнорируется

    println!("{}", visited.contains("Анна")); // true
    println!("{}", visited.contains("Вера")); // false
}
```

В итоге, множество подходит для задач, в которых требуется гарантировать уникальность элементов и эффективно проверять, содержит ли коллекция элемент.

Каждая из них эффективно решает свою задачу:  
- массив — позволяет определить какой элемент лежит под определённым индексом
- хэш-таблица — какое значение связано с ключом

## Задача: есть ли дубликаты? Массив против множества (3–4 мин)

Рассмотрим простую, но показательную задачу: дан массив значений, нужно ответить на вопрос — **есть ли в нём дубликаты?** Другими словами, нужно понять существует ли хотя бы одна пара элементов с одинаковым значением.
### Вариант 1: только массив (наивный подход)

Если у нас есть только массив и никаких дополнительных структур, самый прямой способ — сравнить каждый элемент со всеми, кто идёт после него.

Идея решения:
- для каждого `i` пробежать по всем `j > i`;
- если `arr[i] == arr[j]` — дубликат найден.

Пример на Python:
```python
def has_duplicates_naive(arr):
    n = len(arr)
    for i in range(n):
        for j in range(i + 1, n):
            if arr[i] == arr[j]:
                return True
    return False
```

Проблема такого решения в том, что время работы растёт квадратично: для каждого элемента мы в худшем случае сравниваем его почти со всеми остальными. Это решение имеет асимптотическую сложность `O(n^2)`. На больших массивах оно будет ощутимо тормозить.
### Вариант 2: массив + множество (эффективный подход)
Теперь используем множество. Поскольку множество хранит только уникальные элементы мы можем быстро отвечать на вопрос “присутствует ли этот элемент уже в наборе?”. Средняя сложность операций вставки и проверки вхождения у множества на основе хэш-таблицы `O(1)`.

Идея алгоритма:
- создаём пустое множество `seen`;
- итерируемся по массиву;
- для каждого элемента:
  - если он уже есть в `seen` — мы нашли дубликат;
  - иначе добавляем его в `seen` и идём дальше.

Пример на Python:
```python
def has_duplicates_set(arr):
    seen = set()
    for x in arr:
        if x in seen:
            return True
        seen.add(x)
    return False
```

Теперь время работы — линейное: мы один раз проходим по массиву и для каждого элемента делаем в среднем операции сложностью `O(1)` . Итоговая асимптотика — `O(n)` по времени и `O(n)` по дополнительной памяти, что существенно лучше, чем `O(n^2)` у наивного решения.
### Логика выбора структуры для этой задачи

Сопоставим два подхода, которые мы использовали. Двойной цикл не требует дополнительной памяти, но даёт квадратичное время `O(n^2)`, что плохо масштабируется. С другой стороны, множество требует `O(n)` дополнительной памяти, но даёт линейное время `O(n)`, что хорошо подходит для больших данных.

Поскольку идеальных решений не существует, мы всегда определяем trade-off, который для нас допустим. При решении задачи определите:
- если ключевая операция — **поиск дубликатов** и данные могут быть большими, имеет смысл платить памятью за множество и выигрывать во времени;
- если данных мало или память жёстко ограничена, наивный подход через один массив может оказаться приемлемым, несмотря на `O(n^2)`.
